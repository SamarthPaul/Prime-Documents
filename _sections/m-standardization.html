PR-PD-004

Product Standardization

The Product Development module that documents each entrepreneur's standardisation efforts — recipe / process / packaging SOPs, hygiene, HACCP, GMP, ISO, organic and GI-tag standards — tracks whether existing SOPs are actually being followed, and bridges flagged gaps into a Required pipeline with an implementation status. It carries an entrepreneur-level Task Log and a Resources library of templates and regulatory references.

On-Screen UI Structure

A teal title bar reads Product Standardization with crumb PRIME Rural · Quality & Compliance. Beneath it a horizontal tab rail — Summary · Existing · Required · Resources · Task Log — with the active tab highlighted teal. An entrepreneur chip (avatar + “Auto-fetched from profile”) sits above the tab body and is the implicit parent of every record.

The Existing and Required tabs each hold a list of collapsible standardisation blocks. Each block carries a sea-green header with two inline selectors — Product (fetched from the Product master) and Standardisation Type (fetched from the Standardization Type master) — a status chip, and a Remove button; clicking the header collapses/expands the body. An Existing block's body is a Standardisation Details sub-section (Product Sector, Upload SOP, SOP URL, Developed By, Version, Is-being-followed Yes/No with conditional reason) plus a terracotta Intervention Need panel carrying a toggle that bridges the block into the Required tab.

A Required block's body holds the same details plus Scope for Standardisation and Approved by Core Team checkboxes, a Collaborating Partner field, an optional read-only Justification (from Existing) when bridged, and a terracotta Implementation Status panel (Status select + Notes) that drives the header chip. Bridged blocks show a “↻ Bridged from Existing” marker and lock their Product / Type selectors.

Special widgets: an upload-with-view control and a URL field with a validated “Open” button on every block; the Resources tab is a reference-materials data table (Title / Category / Type / Notes / File); the Task Log tab is an auto-numbered activity table fed by a fixed 19-step task picker. The Summary tab is entirely read-only — a Standardisation-wise overview table, four aggregate cards and a four-tile flags strip. Auto-fetched fields (Product Sector, the bridged Justification) render italic on a cream-deep background and are never editable, with a “Fetched from X” tooltip (Rule 1).

Tab 1: Summary ▼

Entirely read-only. Recalculated on every switch back to Summary. Holds three blocks — a Standardisation-wise overview table, four aggregate cards, and a four-tile Intervention Flags strip. The overview is one row per existing standardisation, with the last two columns pulled from its linked Required (intervention) block; manually-added Required rows are appended below.

Key validations (whole form): the parent record is keyed to one Entrepreneur (unique, required — auto-fetched into the chip); Module is fixed to Product Development and Applicable-To to Product (both read-only). An Existing block must have a Product or a Standardisation Type to count in Summary roll-ups. The “If no, give reasons” field shows only when Being-Followed = No, and Justification shows only when the Intervention toggle is on. Turning the toggle on creates exactly one bridged Required block (Product / Type locked & copied, Justification live-synced); turning it off — or removing the Required block — clears the flag on both sides. SOP URL “Open” / “Resource Open” buttons enable only for valid http(s):// values. One SOP file per block (re-upload replaces). Resources and Task Log require at least one row to remain. All Indian number formatting; no rejected/pending state — status uses the five-state implementation set.
