PR-PD-003

Product Testing

Part of the Product Development module — captures every lab test conducted on an entrepreneur’s product (existing & required), the NABL / accredited lab, the consent letter signed with the entrepreneur, and the full sample-to-result date chain (sample deposited to office → deposited to lab → test results received). Tests that fail a parameter are flagged for retesting, automatically bridging from the Existing tab into the Required tab.
Key behaviour — the Existing → Required intervention bridge. A test conducted is recorded ONCE in the Existing tab. Toggling its “Need Retesting?” flag does not capture any consent or new test data — it only raises a flag and auto-creates a linked card in the Required tab (Product + Test Type locked / sourced from Existing, with the justification carried across read-only). New tests not previously conducted are added manually in the Required tab. The Required tab is where consent management (core-team approval, consent-letter signed toggle + conditional reason, consent letter upload) and the full date chain live; the Existing tab carries only a shorter date chain (deposit → result).

On-Screen UI Structure

Top bar: teal header “Product Testing” with crumb “PRIME Rural · Quality & Compliance”, and an entrepreneur chip (avatar + name) auto-fetched from the profile.
Tab rail: horizontal pill bar with five tabs — Summary · Existing · Required · Resources · Task Log; the active tab renders teal-on-white (Existing opens by default).
Body layout: each tab is a panel. Summary is read-only: a Test-wise Overview table (one row per test), four Aggregate cards (Existing / Cost / Interventions / Implementation Status), and a 4-cell Intervention-flag strip. Existing and Required render each test as a collapsible “test block” (sea-blue header carrying Product + Test Type selectors, a status / flag chip, and a Remove button) with a Test Details sub-section.
Special widgets: (1) Consent toggle — in the Required tab, a “Consent Letter Signed” checkbox (default on) that reveals a conditional “If not signed, reason” textarea when unchecked, plus a Consent Letter upload / view control and a separate “Approved by Core Team” checkbox. (2) Date chain — ordered date fields: Existing = Date of Sample Deposit → Date of Test Results; Required = Date of Sample Deposit to Office → Date of Sample Deposit to Lab → Date of Test Results. (3) Retesting toggle — a terracotta “Intervention Need” panel in Existing with a “Need Retesting?” toggle + Justification, which bridges to Required. (4) Implementation Status selector + chip on each Required card (Pending / In Progress / Completed / On Hold / Not Feasible). (5) File upload / view pills for Testing Report and Consent Letter.
Per Rule 1, every calculated / autofilled field (all Summary aggregates, the overview table cells, bridged Product / Test Type / Justification, the Reference cells) renders italic on a cream-deep background and is non-editable, with a tooltip describing its source / formula.

Tab 3 — Required (tests to be conducted — consent management + full date chain)

Caption: “Tests required — auto-populated from the Existing tab when retesting is flagged, or added manually for new tests to be conducted.” Empty state: “No tests required yet. Flag a test in the Existing tab, or add a new one manually below.” Each card is a collapsible test block added via + Add New Test; header carries Product + Test Type selectors (locked / disabled for bridged cards, title “Locked — sourced from Existing”), a “Bridged from Existing” marker on bridged cards, an Implementation Status chip and a Remove button.

Consent management lives here. Before a required test proceeds, the Fessociate records core-team approval, signs a consent letter with the entrepreneur (toggle), and uploads the signed letter. The consent toggle defaults to ON; un-checking it reveals a mandatory reason. The full date chain (sample → office → lab → result) is also captured on this tab.

Key validations (whole form): The date chain must be strictly ordered everywhere — in Existing, Date of Sample Deposit ≤ Date of Test Results; in Required, Date of Sample Deposit to Office ≤ Date of Sample Deposit to Lab ≤ Date of Test Results (out-of-order dates are rejected). Consent-required rules: when “Consent Letter Signed” is unchecked on a Required test, the “If not signed, reason” field is mandatory; the signed consent letter should be uploaded and core-team approval recorded before a required test proceeds to the lab. A flagged Existing test must carry a Justification, which bridges (read-only) into the auto-created Required card; toggling the flag off removes that card, and toggling it on re-creates it with Product / Test Type locked. Required cards added manually keep Product / Test Type editable. Resources require at least one row. File uploads accept .pdf / .jpg / .jpeg / .png only.
