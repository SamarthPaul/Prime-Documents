PR-OV-006

General Rules (All Forms)

These rules apply to every diagnostic framework form across all modules. Developers must implement these consistently.
Rule 1 — Calculated Fields
Calculated fields must be visually distinct from input fields (italic text, cream-deep background). Each must show its formula in a clean, user-friendly tooltip so field users understand how the value is derived.
Rule 2 — Fetched Fields
Fields auto-fetched from another form must display "Fetched from [X] form" with a clickable redirect link to the source field. The user should be able to see and edit the source if needed.
Rule 3 — Intervention Banner
In the Existing tab, rows with intervention need show the intervention flag in the row banner. In the Required tab, corresponding rows show implementation status in the row banner.
Rule 4 — Product Filters
For all product-level frameworks, show product button filters so users can toggle between products of the entrepreneur. New products added to the portfolio auto-appear in this filter.
Rule 5 — Summary Tab
Summary tab content is generated only after the form is saved for the first time. Before first save, the summary tab shows an empty state with a prompt to save.
Rule 6 — Intervention Details Tab
Intervention Details is its own tab in every framework form — not a section inside the Required tab. It contains Pre-Intervention Comments and Post-Intervention Comments text boxes plus any other intervention-level metadata. Sitting at the framework level (not the row level) means it captures the overall intervention narrative for that entrepreneur in that framework.
Rule 7 — Task Tab Removed
The Task tab shown in some HTML mockups has been removed. Activity logging is handled via the Fellow Activity Logger workspace instead. Framework forms only have the Log Book tab for logging.
Rule 8 — Log Book Structure (per-framework scope)
Every framework has its own Log Book tab. Columns: Date, Fessociate (autofill), Entrepreneur (autofill), Task (dropdown from Framework Task Master, filtered to that framework only), Upload / View (photo), Comments. Scope: the Log Book for framework X shows only activities logged against framework X — whether the activity was added directly inside that framework's Log Book (with framework/entrepreneur prefilled) or created via the Fellow Activity Logger workspace. Activities from other frameworks never appear here. Activity Logger entries also carry through automatically.
Rule 9 — Framework Completion
Profile dashboard shows "Not Started" by default for each framework task. Shows "Completed" once a single activity log exists. If logged again, shows repetition count. Summary popup shows not-started vs completed with percentage.
Rule 10 — Block Restriction
Field users see data from all blocks but can only create/edit for their own assigned block. District and block fields auto-fill from logged-in user's profile when creating new records.
Rule 11 — Cohort-Scoped Monthly Trackers
Any monthly tracker shown anywhere in any framework (any tab) must render month-rows for every year spanning the entrepreneur's cohort start date to cohort end date. The cohort is resolved from the entrepreneur's cohort link on the EP Profile; cohort dates are read from the Cohort master. Example: if the cohort runs Apr 2026 → Mar 2029, the tracker shows 36 month-rows grouped by year. Trackers must never be hard-coded to a single year.
